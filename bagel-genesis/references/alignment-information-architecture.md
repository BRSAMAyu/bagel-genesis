# Alignment Information Architecture

Use this to keep rich alignment information useful without loading irrelevant detail into every agent context.

## Principle

Every agent needs a correct global map, not every local detail.

Use progressive disclosure:

1. **Global capsule:** always small, always loaded by orchestrator.
2. **Domain capsule:** loaded for the relevant product area/module/chapter/research stream.
3. **Task capsule:** generated for a single worker task.
4. **Evidence layer:** loaded only when verifying or resolving conflict.

## Agent-Facing Layers

Use the canonical `.bagel/agent_context/` tree defined in `references/governance-data-model.md`. This file describes how to populate those layers; it is not a second file-tree authority.

### Global Capsule

Keep under 150 lines:

```yaml
global_capsule:
  project_purpose: "..."
  current_phase: "..."
  protected_intent:
    - "..."
  protected_existing_behavior:
    - "..."
  main_conventions:
    - "..."
  active_risks:
    - id: "RISK-..."
      summary: "..."
  context_index: ".bagel/agent_context/context-index.yaml"
  freshness: ".bagel/agent_context/freshness.yaml"
```

### Context Index

Maps topics to files so agents can load only what matters:

```yaml
topics:
  billing:
    capsule: ".bagel/agent_context/domains/billing.md"
    code: ["src/billing/**", "tests/billing/**"]
    reusable_assets: ["src/lib/money.ts"]
    owner: "Project Cartographer"
  onboarding:
    capsule: ".bagel/agent_context/domains/onboarding.md"
    code: ["src/onboarding/**"]
```

### Task Capsule

Generated by Orchestrator for a worker:

```text
Task:
Relevant global facts:
Relevant domain facts:
Existing assets to reuse:
Do not duplicate:
Do not touch:
Known risks:
Allowed files:
Acceptance criteria:
Verification:
Context update obligations:
```

Workers receive task capsules, not the whole project model.

## Human-Facing Layers

Use `.bagel/STATUS.md` as the single human entry point, with the canonical `.bagel/user_briefing/` tree from `references/governance-data-model.md` for deeper layers.

Human layers answer:

- What is this project now?
- How did it get here?
- What did the system decide autonomously?
- What should the user review?
- What risks remain?
- Where can a human go deeper?

## Ownership

| Artifact | Owner | Reviewer |
|---|---|---|
| `global-capsule.yaml` | Orchestrator | Project Cartographer |
| `context-index.yaml` | Project Cartographer | Orchestrator |
| domain capsules | Project Cartographer | relevant reviewer |
| task briefs | Orchestrator | worker reports mismatches |
| user briefing | User Alignment Curator | Orchestrator |
| decision map | Orchestrator | Constitutional Court when needed |
| evolution ledger | Orchestrator | reviewers/auditor |

Workers may propose updates but must not directly rewrite authoritative alignment unless assigned that role.

## Freshness

Maintain `.bagel/agent_context/freshness.yaml`:

```yaml
freshness:
  global_capsule:
    updated_at: "ISO-8601"
    source_changes: ["CHG-..."]
    status: fresh | stale | disputed
  domains:
    billing:
      updated_at: "ISO-8601"
      watched_paths: ["src/billing/**"]
      last_verified_by: "Project Cartographer"
      status: fresh
```

Mark stale when watched files changed, tests/commands changed, user direction changed, or a worker reports mismatch.

## Reliability Rules

- Every non-obvious fact needs evidence link or source change.
- Summaries must distinguish fact, inference, assumption, and decision.
- Stale context must not be used for broad implementation.
- Conflicting context must be resolved before dependent work.
- Task briefs must cite the context capsule versions used.

## Conflict Resolution

When alignment artifacts conflict:

1. Identify the conflict and affected tasks.
2. Stop dependent implementation.
3. Compare evidence: current files/commands, change records, decisions, user instruction.
4. Decide owner:
   - factual project reality → Project Cartographer,
   - user-facing explanation → User Alignment Curator,
   - goal/scope conflict → Orchestrator or Constitutional Court,
   - code behavior conflict → Spec/Code Reviewer.
5. Update authoritative artifact.
6. Add a change record.
7. Resume with a fresh task capsule.

## Update Trigger Matrix

| Trigger | Update |
|---|---|
| module responsibility changes | module-map, domain capsule, context index |
| feature completed/removed | feature inventory, global capsule if important, user current-state |
| new reusable asset | do-not-duplicate, relevant domain capsule |
| command/test changes | project-facts, freshness, user verification docs |
| user instruction changes | decision map, user briefing, constitution if needed |
| recovery/rollback | evolution ledger, risks, current-state, agent context |
| repeated worker confusion | task brief template, domain capsule, context index |
